Memory Boundary For V1 Agent OS

Pick the persistence boundary you want to freeze in the first design

Option A: Project Memory First
UI

Kernel

Long-term memory
project namespace only
Characteristics
faster to ship, simpler mental model, weaker personalization

A. Project-Centric

Only distinguish thread state and project memory. Ignore user-level memory in V1.

Option B: Three-Tier Memory
UI
TUI shell
Kernel
thread state + checkpoints
Control Plane
user memory + project memory + policy-scoped retrieval
Characteristics
clean OS boundary, better future-proofing, more design work now

B. User + Project + Thread

Separate short-term session state from project knowledge and user-level durable preferences from day one.

Recommendation

I recommend Option B: keep retrieval simple in V1, but freeze the storage namespaces now.

Why B is stronger

  • Prevents thread state from becoming fake long-term memory
  • Keeps future personalization separate from repo knowledge
  • Makes approval, policy, and audit logs easier to reason about

Cost of B

  • Need explicit namespace design now
  • Need retrieval rules to avoid over-injecting memory
  • Need memory write policies per scope
