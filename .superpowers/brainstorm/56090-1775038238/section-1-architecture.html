Section 1: System Architecture

Recommended V1 shape: Graph-Centric OS with a local daemon runtime

High-Level Layers
Interface Layer
TUI Shell
input composer / event stream / task list / approval modal / final answer pane
Kernel Layer
SessionKernel
thread manager / command bus / event bus / checkpoint coordinator / interrupt-resume boundary
Control Plane
TaskManager / WorkerManager / PolicyEngine / ApprovalService / MemoryService / ToolRegistry
Execution Runtime
LangGraph root runtime
planner worker / executor worker / verifier worker / background memory worker
Persistence
StoragePort
SQLite adapters for checkpoints / task store / event log / memory namespaces
Primary Flow
User input
↓ TUI emits command
↓ SessionKernel opens or resumes thread
↓ Control Plane resolves task + policy
↓ WorkerRuntime executes graph
↓ Tool calls and approvals emit events
↓ TUI renders stream only
Key Boundary Rules
TUI never mutates runtime state directly

Graphs never bypass policy engine

Workers communicate by events and task contracts

Persistence hides behind StoragePort

Why this shape

  • Keeps LangGraph in its strongest role: orchestration runtime
  • Makes the TUI replaceable later by API or remote UI
  • Supports future daemon split without redesigning the model

Trade-off

  • More abstractions than a simple CLI agent
  • Requires discipline in event and store schema design
  • More upfront architecture work before coding the happy path

Recommendation

Select this if you want the first version to be a real OS-shaped agent kernel rather than a single graph wrapped in a CLI.
