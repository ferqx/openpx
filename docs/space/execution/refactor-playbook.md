# Refactor Playbook

本文档定义 OpenPX 当前阶段进行结构减法的默认玩法。

它的核心目标不是“大重写”，而是：
在不打断主路径的前提下，让复杂文件逐步变薄、边界逐步变清晰。

## 当前默认策略

OpenPX 当前阶段默认使用：

**小步结构减法**

也就是：

1. 先确认主路径和真实职责
2. 先抽支持逻辑
3. 再抽生命周期或桥接层
4. 最后才考虑更大的装配或边界调整

不要从一次性重写开始。

## 什么时候可以开始重构

只有当下面四件事已经明确时，才进入重构：

1. 受影响的主要子系统是什么
2. 受影响的入口点或脚本是什么
3. 哪些测试可以证明没有回归
4. 重构完成后哪些稳定结论必须回写文档

如果这四件事答不清，先补理解，不要动手。

## 推荐切法

### 第一类：支持逻辑切出

优先把这些内容从复杂文件中抽出去：

- 类型归一化
- 小型状态辅助函数
- prompt / payload 组装
- 纯函数工具

这样做的好处是：

- 风险低
- 不容易打断主流程
- 能先把“读代码的噪声”降下来

### 第二类：生命周期切出

当支持逻辑已经抽开后，再切：

- run / task 生命周期推进
- approval 解决路径
- 状态收口逻辑

这一步要紧贴现有状态机，不要顺手重定义概念。

### 第三类：桥接层切出

再往后，才处理：

- graph 调用桥接
- runtime / TUI 命令桥接
- protocol 到 kernel 的分发桥接

桥接层的目标是让主流程更短，而不是重新设计协议。

### 第四类：装配收口

最后才处理：

- context 装配
- service 装配
- kernel 装配
- close / cleanup 顺序

装配层的目标是“让入口更清楚”，不是把所有依赖隐藏到看不见。

## 什么时候不该继续拆

出现下面这些情况时，应先停：

- 当前文件刚刚完成一轮稳定切分
- 继续拆会同时改动主路径和测试基座
- 视图合约或状态合约还没稳定
- 还没把上一轮得到的理解结论回写文档

OpenPX 的结构减法必须允许“阶段性收口”，不能因为文件还大就持续拆到失去上下文。

## 当前仓库里已经验证过的做法

当前已经证明可行的顺序包括：

1. 从 `bootstrap.ts` 先抽支持逻辑
2. 再抽 approval resolution（审批解决）
3. 再抽 run/task lifecycle（运行/任务生命周期）
4. 再抽 graph bridging（图执行桥接）
5. 最后收装配根

以及：

1. 从 `app.tsx` 先抽状态支持逻辑
2. 再抽会话同步层
3. 再抽输入分发层

这说明对 OpenPX 来说，**先薄化，再收边界** 是有效策略。

## 重构后必须做的事

### 1. 运行验证

至少运行与改动面匹配的：

```bash
bun run typecheck
```

以及对应的测试切片。

### 2. 更新复杂度清单

如果某个复杂文件：

- 已经完成一刀
- 优先级改变
- 下轮不该继续优先动

就必须更新 [NOISE_CANDIDATES.md](/Users/chenchao/Code/ai/openpx/NOISE_CANDIDATES.md)。

### 3. 回写稳定结论

如果这次重构确认了：

- 主路径边界
- 模块职责
- 状态流检查点
- 推荐阅读顺序

就必须回写到根级文档或 `docs/space/` 对应索引，不要只停留在对话里。

## 不该做的事

- 不要把“文件很大”当成唯一拆分理由
- 不要一边重构一边发明新的产品术语
- 不要一次跨越太多层同时改
- 不要把技术债文档写成重构愿景大全
- 不要在没有验证的情况下宣称结构已经稳定
