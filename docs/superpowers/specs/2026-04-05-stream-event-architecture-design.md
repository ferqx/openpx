# Stream Event Architecture Design

## Overview

Add phase-level streaming event support to the openpx Agent OS kernel. The kernel emits structured stream events during LangGraph execution, enabling UI layers (CLI TUI, Web, App) to render agent progress progressively. CLI shows phase-level status, Web/App can implement rich progressive rendering.

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Event granularity | Phase-level | thinking, tool_call, text_chunk, done — enough for progressive rendering without token-level event storms |
| Transport | SSE (existing) | Extend current `/events` endpoint, no new ports or protocols |
| Persistence | Ephemeral only | Stream events live in memory ring buffer, not SQLite. `thread.view_updated` is the durable source of truth |
| Approach | Unified StreamEvent bus | Clean API boundary: kernel emits standard events, UI layers consume freely |
| Cross-UI sync | turnId + lastSeq | TUI and Web UI share threads; reconnection recovers stream state from ring buffer |

## Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                        LangGraph                             │
│              graph.streamEvents() (native)                   │
└────────────────────────┬────────────────────────────────────┘
                         │ native events
                         │ (on_chat_model_stream, on_tool_start, ...)
                         ▼
┌─────────────────────────────────────────────────────────────┐
│                  StreamEventAdapter                          │
│  - Aggregate tokens → text_chunk (configurable window)       │
│  - Map on_tool_start → tool_call_started                     │
│  - Map on_chain_start → thinking_started                     │
│  - Assign openpx eventId (ulid) + seq (monotonic)            │
│  - Attach turnId for cross-UI correlation                    │
└────────────────────────┬────────────────────────────────────┘
                         │ StreamEvent (openpx domain events)
                         ▼
┌─────────────────────────────────────────────────────────────┐
│                    SessionKernel                             │
│  streamEvents() → AsyncGenerator<StreamEvent>                │
│    ├── eventBus.publishStream(event) → SSE → all subscribers │
│    └── yield event → handleCommand() sync path               │
│                                                              │
│  handleCommand() (unchanged API):                            │
│    internally uses streamEvents(), collects final result     │
└────────────────────────┬────────────────────────────────────┘
                         │
              ┌──────────┴──────────┐
              ▼                     ▼
┌─────────────────────┐  ┌─────────────────────┐
│   StreamEventBuffer │  │   SSE /events       │
│   (memory ring)     │  │   endpoint          │
│   - per turnId      │  │   ?lastSeq=N        │
│   - 500 events max  │  │   resume + live     │
│   - LRU cleanup     │  └──────────┬──────────┘
└─────────────────────┘             │
                         ┌──────────┴──────────┐
                         ▼                     ▼
                    ┌─────────┐          ┌─────────┐
                    │   TUI   │          │  Web UI │
                    │ phases  │          │progressive│
                    └─────────┘          └─────────┘
```

## StreamEvent Types

```typescript
type StreamEventBase = {
  eventId: string;      // ulid
  threadId: string;
  taskId: string;
  turnId: string;       // correlates events to a conversation turn
  seq: number;          // monotonic sequence number for ordering/dedup
  timestamp: string;    // ISO8601
};

// Phase: Thinking
type ThinkingStartedEvent = StreamEventBase & {
  type: "stream.thinking_started";
  payload: { model: string };
};

type ThinkingChunkEvent = StreamEventBase & {
  type: "stream.thinking_chunk";
  payload: { content: string };
};

// Phase: Tool Call
type ToolCallStartedEvent = StreamEventBase & {
  type: "stream.tool_call_started";
  payload: { toolName: string; toolCallId: string; args: unknown };
};

type ToolCallCompletedEvent = StreamEventBase & {
  type: "stream.tool_call_completed";
  payload: { toolName: string; toolCallId: string; result: unknown; success: boolean };
};

// Phase: Text Response
type TextChunkEvent = StreamEventBase & {
  type: "stream.text_chunk";
  payload: { content: string; index: number };
};

// Phase: Completion
type StreamDoneEvent = StreamEventBase & {
  type: "stream.done";
  payload: { summary: string; status: "completed" | "failed" | "interrupted" };
  // summary is the final accumulated text from all text_chunk events for this turn,
  // generated by the StreamEventAdapter when the root chain ends
};

type StreamEvent =
  | ThinkingStartedEvent
  | ThinkingChunkEvent
  | ToolCallStartedEvent
  | ToolCallCompletedEvent
  | TextChunkEvent
  | StreamDoneEvent;
```

## StreamEventAdapter

Maps LangGraph native `streamEvents()` output to openpx `StreamEvent` types.

```typescript
interface StreamEventAdapter {
  wrapStreamEvents(input: {
    graph: CompiledStateGraph;
    graphInput: Record<string, unknown>;
    config: LangGraphConfig;
    threadId: string;
    taskId: string;
    turnId: string;
  }): AsyncGenerator<StreamEvent, StreamDoneEvent, void>;
}
```

### Token Aggregation

`on_chat_model_stream` fires per token. The adapter aggregates tokens into `text_chunk` events using a configurable strategy:

- **Time window**: emit every N ms (default: 100ms)
- **Content size**: emit every N characters (default: 200 chars)
- Whichever threshold is hit first triggers emission

This reduces event volume while maintaining smooth progressive rendering for Web UI.

### Event Mapping

| LangGraph Event | openpx StreamEvent | Notes |
|---|---|---|
| `on_chat_model_start` | `stream.thinking_started` | Extract model name. Only emitted for the root graph's LLM call (responder mode). In planner mode, the planner's LLM calls produce `text_chunk` directly. |
| `on_chat_model_stream` | `stream.text_chunk` | Aggregated per window. All LLM text output maps here. |
| `on_chat_model_stream` (reasoning) | `stream.thinking_chunk` | If the model supports reasoning/thinking blocks (e.g. Claude extended thinking), these map to `thinking_chunk`. Otherwise, all text goes to `text_chunk`. |
| `on_tool_start` | `stream.tool_call_started` | Extract tool name + args |
| `on_tool_end` | `stream.tool_call_completed` | Extract result + success |
| `on_chain_end` (root) | `stream.done` | Final summary + status. Summary is the final accumulated text from all `text_chunk` events. |

## Kernel Integration

### SessionKernel.streamEvents()

```typescript
class SessionKernel {
  async *streamEvents(command: SubmitInputCommand): AsyncGenerator<StreamEvent> {
    // 1. Create task, assign turnId
    // 2. Call adapter.wrapStreamEvents(...)
    // 3. For each event:
    //    - Publish to eventBus (SSE consumers receive it)
    //    - Yield to caller (handleCommand sync path)
    for await (const event of adapter.wrapStreamEvents(...)) {
      this.eventBus.publishStream(event);
      yield event;
    }
  }

  // Existing API unchanged — internally uses streamEvents()
  async handleCommand(command): Promise<SessionCommandResult> {
    let finalResult: StreamDoneEvent | undefined;
    for await (const event of this.streamEvents(command)) {
      if (event.type === "stream.done") finalResult = event;
    }
    return this.finalize(finalResult);
  }
}
```

### EventBus Extension

```typescript
class EventBus {
  // Existing: durable events (persisted to SQLite)
  publish(event: KernelEvent): void;
  subscribe(handler: (event: KernelEvent) => void): () => void;

  // New: ephemeral stream events (memory only)
  publishStream(event: StreamEvent): void;
  subscribeStream(handler: (event: StreamEvent) => void): () => void;
}
```

## SSE Reconnection & Stream Recovery

### StreamEventBuffer

In-memory ring buffer indexed by `turnId`:

```typescript
class StreamEventBuffer {
  // Per-turn buffer, max 500 events
  push(turnId: string, event: StreamEvent): void;
  resume(turnId: string, lastSeq: number): StreamEvent[];
  cleanup(turnId: string): void;  // called after stream.done
}
```

### SSE Endpoint

```
GET /events?workspaceRoot=...&projectId=...&lastSeq=42
```

- `lastSeq` is optional, client's last received `seq`
- Server first pushes buffered events with `seq > lastSeq`
- Then switches to live streaming
- If buffer miss (cleaned up or seq expired), client receives empty recovery and falls back to `thread.view_updated` for final state

### Reconnection Flow

```
UI disconnects → reconnects
    ↓
GET /events?lastSeq=42
    ↓
1. Push buffered events with seq > 42 (thinking_chunk, text_chunk...)
2. Switch to live stream
3. Push thread.view_updated (final state fallback)
    ↓
UI receives historical events → continues rendering
UI receives final state → confirms completeness
```

### Memory Management

- Per-turn buffer: 500 events max (~50-100KB)
- Global limit: LRU eviction of oldest inactive turns. A turn is "inactive" after `stream.done` is emitted for that turnId.
- Cleanup delay: 5 minutes after `stream.done` to allow reconnection window. After this delay, the turn's buffer is eligible for LRU eviction.
- On memory pressure: evict oldest inactive turns first, never evict active turns

## Cross-UI Thread Synchronization

TUI and Web UI share the same kernel daemon. Threads are cross-UI shared resources.

### Scenario

```
T1: Web UI creates thread-A, sends "analyze project"
T2: Agent starts executing → emits stream.* events
T3: User opens TUI, enters thread-A
    → TUI needs to see:
      a) User message "analyze project"
      b) Ongoing stream events (thinking, tool_call, text_chunk...)
      c) If execution completed, full answer
```

### Snapshot Structure for Reconnection

`hydrateSession()` returns a `RuntimeSnapshot` that includes:

```typescript
type RuntimeSnapshot = {
  // ... existing fields ...
  resumeAnchor: {
    lastEventSeq: number;    // last seq number the kernel has processed
    narrativeRevision: number;
  };
  answers: Array<{
    content: string;
    turnId: string;
    createdAt: string;
  }>;
  activeTask?: {
    taskId: string;
    turnId: string;
    status: string;
    summary: string;
  };
};
```

### TUI Entry Flow

```
TUI enters thread-A
    ↓
1. hydrateSession()
   → Get snapshot: { answers: [...], activeTask: {...}, resumeAnchor: {...} }
   → Build historical message list
    ↓
2. Connect SSE: GET /events?lastSeq=<snapshot.resumeAnchor.lastEventSeq>
   → Push buffered stream events with seq > lastSeq
   → If active turn exists → receives stream.text_chunk etc.
   → If no active turn → receives thread.view_updated (final state)
    ↓
3. Render:
   - Historical messages: display directly
   - In-progress messages: reconstruct from stream events
   - New messages: receive and render in real-time
```

### RuntimeClient API

```typescript
class RuntimeClient {
  // Existing
  subscribeEvents(): AsyncIterable<EventEnvelope>;

  // New: stream event filter with optional resume
  subscribeStreamEvents(options?: { lastSeq?: number }): AsyncIterable<StreamEvent>;
}
```

`subscribeStreamEvents()` implementation:
- If `lastSeq` provided, appends `?lastSeq=N` to the SSE URL
- Filters incoming events by `type.startsWith("stream.")`
- Reconnects automatically on disconnect, preserving `lastSeq` from last received event

### Message Construction

```typescript
// TUI enters thread:
// 1. hydrateSession() → get historical messages
const history = snapshot.answers.map(a => ({
  role: "assistant",
  content: a.content,
  turnId: a.turnId,
  status: "completed"
}));

// 2. Connect SSE + lastSeq → get ongoing stream
for await (const event of client.subscribeStreamEvents({ lastSeq })) {
  if (event.type === "stream.text_chunk") {
    // If turnId exists → append content
    // If turnId doesn't exist → create new message (in-progress)
    upsertMessage(event.turnId, event.payload.content);
  }
  if (event.type === "stream.done") {
    markMessageComplete(event.turnId);
  }
}
```

## UI Layer Behavior

### TUI (CLI) Phase Rendering

TUI renders agent progress by phases, not token-by-token:

```
❯ 分析当前项目结构

  ● Thinking... (2.1s)
  ◌ Reading: package.json...
  ✔ Read package.json
  ◌ Listing files...
  ✔ Listing files

  Agent: 当前项目是一个 Agent OS，基于 LangGraph 构建。
         核心模块包括 kernel、runtime、control...
         (text blocks appear progressively, not character-by-character)

────────────────────────────────────────────────────────────
❯ Describe the task or ask a question...
```

**StreamPhase tracking in TUI:**

```typescript
type StreamPhase =
  | { type: "thinking"; startedAt: number }
  | { type: "tool_call"; toolName: string; status: "running" | "done" }
  | { type: "text"; content: string };

// On stream.* event → append phase
// TUI renders all phases in order
```

### Web UI (Ready for Implementation)

Kernel provides stream events; Web UI implements its own rendering:

```typescript
// Web UI pseudo-code
for await (const event of client.subscribeStreamEvents()) {
  switch (event.type) {
    case "stream.thinking_started":
      showThinkingPhase();
      break;
    case "stream.text_chunk":
      appendToAssistantMessage(event.payload.content);  // progressive
      break;
    case "stream.tool_call_started":
      showToolCallSpinner(event.payload.toolName);
      break;
    case "stream.done":
      finalizeMessage(event.payload.summary);
      break;
  }
}
```

Web UI can implement:
- Typewriter effect for text chunks
- Expandable thinking panels
- Interactive tool call cards
- Real-time progress indicators

### TUI vs Web UI Comparison

| Capability | TUI | Web UI |
|---|---|---|
| Thinking phase | "Thinking..." text | Expandable chain-of-thought panel |
| Tool calls | Single-line status | Interactive tool call cards |
| Text response | Full text blocks appear | Typewriter effect, character-by-character |
| Reconnection | Rebuild from phase array | Rebuild from stream events |

## Backward Compatibility

| Existing API | Behavior | Change |
|---|---|---|
| `handleCommand()` | Returns complete result | Internally uses `streamEvents()`, externally unchanged |
| `thread.view_updated` | Pushed after completion | Unchanged |
| `subscribeEvents()` | Pushes all events | Now includes `stream.*` events |
| TUI rendering | Shows complete answer | Now shows phase-level progress |
| ModelGateway | Internal stream accumulation | Now emits stream events through adapter |

2. **Token aggregation tests** — verify time window and size window triggers
3. **StreamEventBuffer tests** — verify push, resume, cleanup, LRU eviction
4. **SessionKernel integration tests** — verify `streamEvents()` yields events in order
5. **SSE reconnection tests** — verify `lastSeq` recovery and buffer hit/miss
6. **Cross-UI sync tests** — verify TUI can recover ongoing stream from Web UI session
7. **TUI rendering tests** — verify phase-level rendering without token-by-token updates
8. **Backward compatibility tests** — verify `handleCommand()` still returns complete result
